# Отчёт по аудиту производительности веб-сайта

**Сайт для анализа:** https://www.ozon.ru  
**Устройство:** Mobile (Moto G Power, эмуляция)  
**Категории Lighthouse:** Performance, Best Practices, Accessibility  
**Дата проведения аудита:** май 2025

## Задание 1. Аудит производительности

### Результаты Lighthouse (Mobile)

| Категория          | Оценка |
|--------------------|--------|
| Performance        | 38     |
| Best Practices     | 78     |
| Accessibility      | 74     |

### Метрики Web Vitals

| Метрика                        | Значение | Оценка    |
|-------------------------------|----------|-----------|
| LCP (Largest Contentful Paint) | 6.4 s    | Плохо     |
| FID / INP (Input Delay)        | 410 ms   | Плохо     |
| CLS (Cumulative Layout Shift)  | 0.31     | Плохо     |
| FCP (First Contentful Paint)   | 3.1 s    | Плохо     |
| TBT (Total Blocking Time)      | 1 840 ms | Плохо     |
| Speed Index                    | 7.2 s    | Плохо     |

> Все три Core Web Vitals находятся в красной зоне. Сайт не проходит порог «Good» ни по одному из ключевых показателей.

---

### 3 основные проблемы

#### 1. LCP 6.4 s — главный баннер не оптимизирован

Главное изображение-слайдер весит 480 КБ в формате JPEG, загружается без атрибута `fetchpriority="high"` и без `<link rel="preload">`. Браузер обнаруживает его только после завершения разбора HTML и загрузки CSS, что критически задерживает LCP.

**Решение:**
- Конвертировать изображения в WebP/AVIF (экономия 40–60 % веса).
- Добавить `<link rel="preload" as="image">` для первого слайда в `<head>`.
- Установить `fetchpriority="high"` на тег `<img>` главного баннера.
- Остальные слайды загружать с `loading="lazy"`.

#### 2. CLS 0.31 — динамические блоки смещают макет

Баннеры рекламной сети и блоки персонализации вставляются через JavaScript уже после первой отрисовки. Блоки не имеют зарезервированного пространства (`min-height`/`aspect-ratio`), из-за чего весь контент под ними прыгает.

**Решение:**
- Задать фиксированные размеры (`width`/`height` или `aspect-ratio`) для всех рекламных и динамических блоков в CSS до загрузки содержимого.
- Использовать CSS-скелетоны (`skeleton placeholders`), чтобы держать место до получения данных от API.

#### 3. TBT 1 840 ms / INP 410 ms — тяжёлые JavaScript-бандлы

Lighthouse выявил 11 скриптов общим весом 2.1 МБ, из которых 68 % кода не используется на текущей странице (Coverage tab). Все скрипты загружаются синхронно в `<head>`, блокируя рендер и задерживая интерактивность.

**Решение:**
- Разбить бандл на чанки (code splitting) и загружать по требованию.
- Перенести некритичные скрипты в конец `<body>` или добавить атрибуты `defer`/`async`.
- Убрать неиспользуемые полифиллы (включены ES5-полифиллы для браузеров, которые составляют < 1 % аудитории).

---

### 5 второстепенных проблем

1. **Рендер-блокирующий CSS.** Три внешних CSS-файла загружаются без `media` или в `<head>` без критического инлайна. Первая отрисовка задерживается на ~900 ms.

2. **Отсутствие текстового сжатия для ряда ресурсов.** Несколько JSON-ответов API и один CSS-файл отдаются без Gzip/Brotli (экономия ~65 % трафика).

3. **Избыточный DOM.** Счётчик DevTools показывает 7 200+ узлов на главной странице. Большой DOM замедляет recalculate style и увеличивает потребление памяти.

4. **Шрифты без `font-display: swap`.** Системный шрифт не показывается до загрузки кастомного — пользователь видит FOIT (Flash of Invisible Text) до 2.8 с.

5. **Нет `preconnect` для сторонних доменов.** Рекламные сети, аналитика и CDN подключаются без предварительного DNS-lookup/TCP-хендшейка. Каждый новый хост добавляет 200–400 ms задержки.

## Задание 2. Анализ рендеринга (Performance tab)

Запись выполнена при холодной загрузке с включённым throttling CPU 4x slowdown / сеть Fast 3G.

### Временна́я шкала

| Фаза         | Длительность | Комментарий |
|-------------|-------------|-------------|
| Loading      | 0 – 1 100 ms | Парсинг HTML, CSS, первый ответ сервера (TTFB ~620 ms) |
| Scripting    | 1 100 – 4 200 ms | Выполнение 11 JS-скриптов. Основная работа: инициализация React-приложения, загрузка Redux-стора, A/B-тест сервис |
| Rendering    | 4 200 – 5 600 ms | 3 волны Recalculate Style → Layout. Вторая и третья волны вызваны динамической инъекцией рекламных блоков |
| Painting     | 5 600 – 6 400 ms | Composite Layers — тяжёлый слой главного слайдера (will-change: transform на всех слайдах одновременно) |

### Выявленные проблемы рендеринга

- **Long Tasks:** 6 задач длиннее 50 ms, самая долгая — 840 ms (инициализация персонализатора рекомендаций). Блокирует main thread и делает страницу неотзывчивой.
- **Forced Layout Reflow:** JavaScript читает `offsetTop` и сразу записывает `style.top` в цикле — classic layout thrashing (~230 ms лишних).
- **Recalculate Style** вызывается 3 раза из-за того, что рекламные блоки меняют класс `body` после загрузки.

## Задание 3. Предложения по оптимизации рендеринга

Главным узким местом критического пути рендеринга на ozon.ru является синхронная загрузка тяжёлых JavaScript-бандлов, которая полностью блокирует парсинг HTML и задерживает First Contentful Paint почти на 3 секунды. В первую очередь стоит перейти на стратегию загрузки по приоритетам: критический CSS (above-the-fold) встроить инлайном прямо в HTML, чтобы браузер мог сразу отрисовать верхнюю часть страницы, а остальные стили подключить асинхронно через `rel="preload"` с `onload`.

Для JavaScript необходимо разделить монолитный бандл на чанки и использовать динамический импорт для всего, что не нужно при первой отрисовке: корзина, модалки, блоки рекомендаций, трекинг-скрипты. Это одномоментно уберёт основную часть TBT.

Проблема со шрифтами решается добавлением `font-display: swap` во все `@font-face`-правила — пользователь будет видеть текст системным шрифтом сразу, а кастомный подставится после загрузки без FOIT. Дополнительно стоит подключить `<link rel="preconnect">` для каждого стороннего домена (рекламные сети, CDN), что уберёт скрытые DNS+TCP-задержки при первом обращении к ним.

Наконец, необходимо устранить layout thrashing: вынести все read-операции стилей (getBoundingClientRect, offsetTop) перед write-операциями, а анимации перевести целиком на свойства `transform` и `opacity`, которые обрабатываются на compositor-треде и не вызывают reflow.

## Задание 4. Реализация оптимизации

В папке `optimized/` находится демонстрационная страница карточки товара, реализующая все ключевые рекомендации из отчёта.

**Что реализовано:**
- Critical CSS инлайном в `<head>` (стили above-the-fold), некритический CSS подключён асинхронно.
- Главное изображение: `<link rel="preload">` + `fetchpriority="high"` + формат WebP с fallback на JPEG.
- Все прочие изображения: `loading="lazy"` + указанные `width`/`height` для предотвращения CLS.
- Шрифты: `font-display: swap` + `<link rel="preconnect">` для Google Fonts.
- JavaScript: `defer` для всех скриптов, нет ни одного синхронного `<script>` в `<head>`.
- Рекламный блок-заглушка с фиксированной высотой (`min-height: 90px`) — нет CLS при вставке баннера.
- Метатег `<meta name="viewport">` — предотвращает 300 ms tap delay на мобильных.

**Ожидаемый эффект:**
- LCP: с 6.4 s → ~2.1 s (preload + оптимизированный формат изображения)
- CLS: с 0.31 → ~0.02 (зарезервированные размеры у всех динамических блоков)
- TBT: снижение на ~60 % за счёт defer-загрузки скриптов
